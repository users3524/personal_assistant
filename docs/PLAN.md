## 5. Phase B：数据捕获最小闭环

优先落地白天低成本捕获：

1. 新增对话 turn 记录与每日 15 轮限制。
2. 输入框限制 500 字，STT 限制 60 秒。
3. 高优任务完成后触发单轮追问。
4. 熔断后进入离线便签模式。
5. 白天 Prompt 只含当天上下文和当天待办，不查历史。

## 6. Phase C：待办四层级完善

1. UI 层明确“分类 -> 清单 -> 父任务 -> 子任务”。
2. Repository 层批量 IN 查询子任务，消除 N+1。
3. 父任务完成、取消、软删除必须事务级联。
4. 仪表盘统计补充单元测试，锁定 `parent_id IS NULL`。
5. 所有底层任务绑定人生罗盘五维之一。

## 7. Phase D：深夜日报引擎

1. 定时任务只在 2:00-5:00、充电、Wi-Fi 下运行。
2. 不满足条件则顺延到次日首次打开 App 后低优先级执行。
3. 打包 Todo、文玩、习惯、对话、便签为结构化 JSON。
4. 原始文本裁剪到 8000 字以内。
5. 使用结构化输出生成 Markdown 日报与高光 JSON。
6. Parser 失败最多重试 2 次，失败后标记待手动校准。

## 8. Phase E：长效记忆与人生罗盘

1. 深度日报向量化为 1536 维切片。
2. 周报和规划纠偏只检索 Top-K=5。
3. 每条记忆切片最多 400 字。
4. 人生罗盘固定 5 个维度。
5. 长期目标 30 天内仅允许修改一次。

## 9. Phase F：高光与简历

1. 日报自动高光单日最多 2 条。
2. 本地筛选 `#简历素材` 标签与时间范围。
3. STAR 润色最多输出 3 条 bullet。
4. 模型只输出纯文本，不参与样式。
5. Flutter 模板负责 A4 / 页面 / 导出排版。

## 10. 协作约定

1. 后续每次对话结束都提交一次本轮已完成变更。
2. 提交前先检查 `git status`，避免混入用户已有改动。
3. 若只改文档，提交范围限定为 `docs/`。
4. 业务代码修改阶段必须先对照本设计文档。
5. 文玩记录模块默认保留，任何移除都需要再次确认。
