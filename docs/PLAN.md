# 实施计划

最后更新：2026-06-21

本计划是重新整体审视后的执行顺序。核心原则：先让已有底座产生用户可见价值，再继续扩展 AI 能力。

## 1. 当前阶段结论

| 范围 | 结论 |
| --- | --- |
| 产品主线 | 应收敛到“行动/兴趣素材 -> 复盘 -> 高光 -> 简历/长期记忆” |
| 数据库 | `schemaVersion = 15`，20 张表，AI 底座已明显超前 |
| 待办 | 已能承载行动事实；下一步不是重做待办，而是改善清单、标签/重复曝光和高光来源衔接 |
| 文玩 | 功能足够丰富；应停止估值扩张，把打卡备注/照片摘要作为复盘素材 |
| AI 复盘 | 已有入口、日/周报、成本闸门、生成任务表、结构化 runner；缺真实组包和写入闭环 |
| 简历 | 编辑和 PNG 导出可用；缺 PDF、高光投递和 STAR 入口 |
| 安全备份 | 覆盖面足够，但明文覆盖导入风险变大 |

## 2. 阶段目标

### Phase 0：文档与口径收敛

目标：让文档重新反映当前项目，并给出下一阶段 backlog。

验收：

1. 所有 docs 文件都更新到同一产品主线。
2. TODO 从“历史完成清单”改为“下一阶段任务”。
3. 文档不再把已落地底座写成未设计，也不把未完成闭环写成已实现。

### Phase 1：复盘生成闭环

目标：让 `review_generation_jobs` 从状态表变成真实可执行任务。

交付切片：

1. Raw context pack builder：从待办、chat_turns、已有日报草稿、patting_logs 组包。
2. 前台手动/自动补偿执行：pending job 可调用 `NightlyStructuredReviewRunner`。
3. 写入日报：成功写入 `daily_reviews`，失败标记 `calibration_required`。
4. 用户可见状态：复盘首页能看到待生成、失败、需校准状态。
5. WorkManager 回调只调用同一执行入口，不单独写另一套逻辑。

不做：

1. 不接 RAG。
2. 不做人生罗盘 UI。
3. 不承诺后台准点生成。

### Phase 2：高光确认与简历投递

目标：让 `milestones` 变成用户可管理的证据池。

交付切片：

1. 自动候选：基于日报/待办/文玩素材生成最多 0-2 个候选。
2. 高光收件箱：用户确认、忽略、编辑标题/描述/重要性。
3. 来源追溯：能从高光跳回日报、待办或文玩打卡。
4. 项目绑定：项目经历可绑定多个高光并排序。
5. 简历投递：把确认高光转为项目关键交付或 STAR 候选。

不做：

1. 不让 AI 直接改简历排版。
2. 不自动把未确认高光写入正式简历。

### Phase 3：简历输出补齐

目标：让简历从“编辑器 + PNG”升级为可靠输出工具。

交付切片：

1. `ResumePdfExportService`，使用 `pdf` / `printing`。
2. A4 模板、字体策略、分页保护和极端文本处理。
3. PDF 测试：字节生成、A4 尺寸、关键文本、极端文本不抛错。
4. STAR bullet 入口：只基于本地事实生成，最多 3 条。
5. AI 输出清洗接入 UI 状态层。

不做：

1. 不让模型输出 HTML/CSS/Markdown 排版。
2. 不做像素绝对一致的脆弱 golden。

### Phase 4：长期记忆与 RAG

目标：在已有向量表基础上做可控检索，而不是把全量历史灌进 prompt。

交付切片：

1. embedding 生成任务和重建入口。
2. 向量索引状态 UI：缺失、不兼容、需重建、可搜索。
3. 周报/规划 RAG 限窗：Top-K <= 5、单片 <= 400 字、总 prompt 预算受控。
4. 只对确认高光、日报摘要等高价值文本建索引。
5. 大规模后再启用分区和 Isolate 策略。

不做：

1. 不索引图片 base64、API Key、备份路径和原始长 dump。
2. 不在白天对话读取历史 RAG。

### Phase 5：人生罗盘与待办重构

目标：等复盘和高光闭环稳定后，再把待办升级为长期目标系统。

交付切片：

1. 固定五维目标持久化。
2. 30 天修改冷却 UI。
3. 存量根任务迁移审核。
4. 待办清单、标签、重复策略表单补齐。
5. 统计口径继续只以根任务为大盘分母。

## 3. 横向工程任务

| 任务 | 为什么重要 | 优先级 |
| --- | --- | --- |
| 备份 manifest 和预检 | 当前覆盖导入风险高，失败时难以解释 | P0 |
| 可选加密备份 | 数据和图片越来越敏感 | P1 |
| 替换占位 app/widget 测试 | 当前冒烟测试没有真实保护力 | P1 |
| 真机验证 secure storage | Web/Windows 安全强度依赖插件实现 | P1 |
| 旧绝对图片路径迁移工具 | 图片路径已收敛，但旧数据仍可能破图 | P2 |
| 估值物理删除评估 | 不是产品阻塞，只能独立迁移 | P3 |

## 4. 实现顺序规则

1. 每个阶段先写测试，再接 UI，再提交文档同步。
2. 涉及 schema、备份、AI 调用上限、导出文件的改动必须有专项测试。
3. 同一阶段内优先完成一个可演示闭环，而不是并行铺多个半成品底座。
4. 所有 AI 输出都必须经过事实约束、长度限制和用户可见降级。
5. `pageview/` 仍视为无关未跟踪现场，不纳入提交。
